ABOUT -- STAR WARS: DARK FORCES MUSIC / GENERAL MIDI / MT-32 / ADLIB / SOUNDBLASTER

One iMUSE score, four original devices.

Star Wars: Dark Forces (LucasArts, 1995) drove its music with the iMUSE system. Every cue lives in SOUNDS.GOB as a .GMD file — a LucasArts wrapper around a standard MIDI file, sprinkled with private 0x7D SysEx commands that iMUSE used for loops, jumps and gameplay transitions. The original setup menu let you pick a sound device, and the same underlying score was re-interpreted by a different driver for each one. This jukebox plays four of those device paths side by side: General MIDI (Roland SC-55), Roland MT-32, AdLib (OPL2 FM) and Sound Blaster (OPL3 FM). Use the Gen MIDI / MT-32 / AdLib / SoundBlaster filter to switch.

General MIDI: the Sound Canvas mix (premium path)

The General-MIDI path (GMIDI.MDR in the original game) sends the score straight to a Roland SC-55 / SCC-1 Sound Canvas. It is fully multi-timbral — up to sixteen parts at once with standard GM instruments and a dedicated drum channel — so it is the cleanest, fullest version Dark Forces offers. For each cue we take track 0 of the embedded MIDI (the canonical full sequence), strip the private iMUSE control SysEx, and render it through a cycle-accurate WebAssembly Nuked-SC55.

Roland MT-32: a clever, constrained adaptation

The MT-32 path (ROLAND.MDR) is not just “General MIDI on a Roland.” A real MT-32 / LAPC-I has only 8 melodic parts plus one rhythm part, but the Dark Forces score uses around a dozen logical channels. The driver solves this with dynamic part-stealing: it keeps state for all 16 logical channels, but assigns only 8 physical MT-32 parts at any moment. When a new channel needs to play and all 8 parts are busy, it steals the least-recently-used part — sending All Notes Off, then restoring program, volume, pan and pitch-bend for the new channel before the note. Percussion has its own quirks (high notes dropped, a couple of drums remapped). The result is a faithful but lossy re-creation — exactly what the hardware option sounded like in 1995.

We reproduce that algorithm here, mapping GM programs to MT-32 patches with the program table recovered from ROLAND.MDR, and play the result through the Munt CM-32L emulator shared with the TFX jukebox.

AdLib & Sound Blaster: the authentic FM, byte for byte

The two FM builds are not a generic GM-on-OPL approximation — they are the game's own synthesis. We run the original 32-bit DOS FM drivers, FM2.MDR (2-operator OPL2, AdLib) and FM4.MDR (4-operator OPL3, Sound Blaster), inside a CPU emulator. Acting as the iMUSE sequencer, we feed each cue's MIDI events — program changes, note-ons and offs, controllers, pitch bend — straight into the real driver, and capture the exact stream of OPL register writes it produces, including its own instrument bank, voice allocation, key-scaling and percussion handling. That register log is replayed through a cycle-accurate Nuked-OPL3 emulator, so the result is sample-for-sample what the original driver would have sent to a real AdLib or Sound Blaster card in 1995. The OPL3 build additionally uses the chip's second register array (18 channels / 4-operator voices) for a thicker, wider sound than plain AdLib.

How this works

The chain from the 1995 game data to your headphones:

SOUNDS.GOB .GMD (iMUSE / MIDI)-> track-0 SMF-> GM / MT-32 fold / real FM2.MDR + FM4.MDR drivers-> JavaScript timing-> Nuked-SC55 (GM) / Munt CM-32L (MT-32) / Nuked-OPL3 (AdLib + SoundBlaster)-> synthesized sound-> modern headphones

The synthesis engine, ROM images and UI are shared with the sister TFX jukebox — only the iMUSE converters, the FM-driver emulator and this page are new.
